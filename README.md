# AI-B1-3


[프로젝트 1] 자동화 도구 비교 구현 보고서 (최종·줄글 버전)
Make vs Zapier — 동일 워크플로우 실제 구현 비교 분석
주제: 여름철 피서지 선호도 설문 자동 알림 시스템 사용 도구: Make (Integromat), Zapier 작성 기간: 2026년 7월 13일 ~ 7월 14일
________________________________________
1. 과제 개요
본 보고서는 동일한 자동화 워크플로우, 즉 “Google Sheets 설문 응답을 감지하여 선호도 점수가 4점 이상인 경우에만 Slack과 Gmail로 알림을 보내는 자동화”를 서로 다른 두 노코드 도구인 Make와 Zapier로 각각 실제 구현하고, 그 과정과 결과를 비교·분석한 것이다. 과제 요구사항은 Trigger 1개 이상, Action 2개 이상, 조건 분기(Filter) 1개 이상을 포함하되 각 분기 경로가 실제로 최소 1회 이상 실행된 결과를 확인할 수 있어야 한다는 것이었으며, 설계 문서만으로는 인정되지 않고 실제로 동작하는 워크플로우여야 했다. 두 프로젝트 모두 “올여름 가고 싶은 곳은 어디인가요?”라는 동일한 설문 데이터를 사용하였으며, 공통 워크플로우 구조는 Google Sheets에서 새 응답을 감지하면 선호도 점수에 “4점” 또는 “5점”이 포함된 경우만 필터를 통과시키고, 이후 Router 혹은 순차 연결을 통해 Slack 메시지와 Gmail 메일이라는 두 액션으로 분기되는 형태였다. 과제 수행 과정에서는 GPT를 통해 요구사항 해석과 전체 작업 순서를 설계하였고, Gemini로부터 Make 화면 캡처를 기반으로 한 단계별 클릭 가이드를 받았으며, Claude와의 대화를 통해 설문지 초안을 작성하고 실행 오류를 진단·검증하였다.
2. 설문지 설계
자동화의 입력 데이터를 만들기 위해 Google Form으로 “올여름 가고 싶은 곳은 어디인가요?”라는 제목의 설문지를 제작하였다. 설문은 총 7개 문항으로 구성되었으며, 세션 1에서는 이름·연령대·성별과 같은 기본 정보를 수집하고, 세션 2에서는 응답자가 실외와 실내 중 어느 쪽을 선호하는지를 분류한 뒤, 세션 3·4에서는 그 분류에 따라 실외 또는 실내 세부 장소를 선택하도록 하였다. 마지막 세션 5의 공통 질문에서는 가고 싶은 구체적인 장소와 그 이유를 서술형으로 입력받고, 마지막 7번 문항에서 해당 장소에 대한 선호도를 1점(매우 낮음)부터 5점(매우 높음)까지 5단계로 선택하도록 하여, 이 점수를 조건 분기의 기준으로 사용할 수 있게 설계하였다. 설문 응답은 총 10건이 수집되었으며, Google Sheets에 자동으로 기록되도록 연동하였다.
3. 전체 작업 순서
과제는 이틀에 걸쳐 진행되었다. 7월 13일에는 Make를 이용해 Google Sheets Trigger, Filter, Slack Action으로 이루어진 기본 워크플로우를 완성하였다. 순서상으로는 먼저 Make에서 새 Scenario를 생성하고, Google Sheets Trigger(Watch New Rows)를 연결한 뒤, 선호도 4점 이상만 통과시키는 Filter를 설정하고, Slack의 Send a Message 액션을 추가하여 Run once로 테스트하는 과정을 거쳤다. 이 과정에서 Drive 값 누락, 트리거 Limit값 제한, 필터 OR 조건 누락, 트리거 커서 미초기화라는 네 가지 오류를 순차적으로 진단하고 해결하였으며, 최종적으로 테스트 응답을 추가하고 커서를 초기화하여 결과를 캡처하였다.
7월 14일에는 두 갈래로 나뉘어 작업이 진행되었다. 먼저 Make 쪽에서는 과제 요구사항인 “Action 2개 이상”을 충족하기 위해 Router를 추가하고 두 번째 액션으로 Gmail의 Send an email을 병렬로 연결하였는데, 이 과정에서 Gmail 계정이 아직 연동되지 않아 발생한 Connection 오류를 계정 재연동으로 해결하였다. 이후 필터가 Router 앞에 정상적으로 위치해 있는지 재조정 체크를 거쳐 커서를 다시 초기화한 뒤 최종 테스트를 진행하고 Make·Slack·Gmail의 결과를 모두 캡처하였다.
같은 날 Zapier 쪽에서는 처음 사용하는 도구였기 때문에 계정 가입부터 시작하였다. Gmail, Google Sheets, Slack을 자주 사용할 앱으로 선택한 뒤, Zapier의 Copilot 기능에 “새 행이 Google Sheets에 추가되면 Slack 메시지와 Gmail을 통한 이메일을 보내라”는 자연어 프롬프트 한 줄을 입력하여 Trigger 1개와 Action 2개(Gmail, Slack)로 구성된 3단계 Zap 초안을 자동으로 생성받았다. 이후 세 앱의 계정을 각각 연동하는 과정에서 연결 만료 오류가 발생해 재연동하였고, 초안을 그대로 게시한 뒤 테스트한 결과 신규 행 1건만 처리되는 현상을 확인하였다. 기존에 쌓여 있던 데이터를 일괄 처리할 수 있는 방법을 검토하였으나 관련 기능이 무료 요금제에서 제한되어 있음을 확인하고, 과제 요구사항을 재확인한 뒤 “신규 응답 자동 알림” 구조로 방향을 확정하였다. 이후 Copilot 초안에는 없던 Filter by Zapier 단계를 Google Sheets와 Gmail 사이에 추가하여 조건 분기를 구현하고, Gmail·Slack 액션의 필드를 재배열·재설정한 뒤 재게시하여 새로운 설문 응답으로 최종 테스트를 진행하고 결과를 캡처하였다.
4. Make 구현 결과
Make로 완성한 최종 워크플로우는 Google Sheets의 Watch New Rows 트리거로 새 행을 감지하고, “4점 이상만 통과”라는 라벨의 필터가 Router 앞에 위치하여 두 액션 모두에 동일한 조건이 적용되도록 하였으며, Router를 거쳐 Slack의 Send a Message와 Gmail의 Send an email 두 갈래로 분기되는 구조였다. 실제 실행 결과 화면에서는 Google Sheets 10건이 처리되고 Filter를 10건 모두 통과하여 Router 분기 이후 Slack 10건, Gmail 10건이 모두 초록색 체크 표시와 함께 정상 전송된 것을 확인할 수 있었다. 이는 Slack 채널에 실제로 도착한 알림 메시지 화면과 Gmail 수신함에 도착한 “[과제] 4점 이상 설문 데이터가 접수되었습니다.”라는 제목의 메일 화면으로 각각 증빙되었다.
Make 구현 과정에서 겪은 트러블슈팅은 크게 네 가지였다. 첫째, Run once 테스트 시 Google Sheets 모듈에서 Drive 값이 비어 있다는 오류가 발생하였는데, 이는 Drive 항목을 다시 “My Drive”로 선택하고 Spreadsheet ID와 Sheet Name을 재확인하여 해결하였다. 둘째, 전체 응답 10건 중 4점 이상 응답이 9건이었음에도 실제로는 1건만 처리되는 불일치가 발견되었는데, 원인은 트리거의 Limit 값이 기본값인 1로 제한되어 있었기 때문이며 이를 100으로 넉넉하게 수정하여 해결하였다. 셋째, Limit을 수정한 뒤에도 통과 건수가 3건에 그쳐 재검토한 결과 필터 조건에 “4점 포함” 조건만 남아 있고 “5점 포함”이라는 OR 조건이 누락되어 있었음을 확인하고 Add OR rule로 두 번째 조건을 추가하여 해결하였다. 넷째, 필터 수정 직후 재실행 결과가 0건으로 나왔는데, 이는 Watch New Rows 트리거가 이전 실행에서 이미 전체 응답을 한 번 읽어 커서가 끝까지 이동해 있었기 때문이며, Choose where to start 설정을 다시 열어 All로 재설정함으로써 커서를 초기화하여 해결하였다.
5. Zapier 구현 결과
Zapier로 완성한 최종 Zap은 Google Sheets의 New Spreadsheet Row 트리거로 새 행을 감지하고, Filter by Zapier가 7번 문항의 값에 “4점” 또는 “5점”이 포함되는 경우만 통과시킨 뒤, Gmail의 Send Email과 Slack의 Send Channel Message 순서로 순차 연결되는 구조였다. Zap Runs 로그에서는 필터 단계에 “Your run passed the rules for this filter.”라는 문구가 표시되어 필터를 통과한 데이터만 두 액션으로 정상 전달되었음을 확인할 수 있었으며, 새로운 설문 응답을 제출한 뒤 Trigger부터 Filter, Gmail, Slack까지 모든 단계가 정상 처리된 결과 화면과 함께, Slack 채널과 Gmail 수신함에 실제로 도착한 알림을 각각 캡처로 증빙하였다.
Zapier 구현 과정에서는 총 여섯 가지 이슈를 겪었다. 첫째, Google Sheets 계정을 최초 연동한 뒤 트리거 설정 화면에서 “This account connection is expired.”라는 오류가 표시되었는데, 트리거 모듈에서 Reconnect this account를 눌러 Google 계정을 재로그인함으로써 해결하였다. 둘째, 기존에 쌓여 있던 응답 11건 중 4점 이상 응답이 다수였음에도 신규 테스트 시 1건만 Slack/Gmail로 전송되었는데, 이는 New Spreadsheet Row 트리거가 새로 추가된 행 1개만 가져오는 구조이기 때문이며, 새 테스트 응답을 직접 추가해 봄으로써 이것이 정상 동작임을 확인하였다. 셋째, 기존 데이터를 한 번에 처리하고 싶었으나 Looping by Zapier, Digest by Zapier, Transfer data와 같은 기능들이 모두 무료(체험) 요금제에서 제한되어 있음을 GPT와의 대화를 통해 확인하고, 과제 지시서를 재확인한 결과 “기존 데이터 전체 전송”이 요구사항이 아님을 확인하여 “신규 응답 자동 알림” 구조로 방향을 조정하였다. 넷째, Zap history에서 과거 실행 기록을 조회하려 했으나 “Can’t access this run” 오류가 발생하여, 대신 Zap Runs 로그와 Run details에서 개별 실행 데이터를 직접 확인하는 방식으로 대응하였다. 다섯째, Transfer data 기능을 시도하였으나 별도의 Zapier Table 생성이 선행되어야 하는 유료·고급 기능임을 확인하고 과제 범위를 벗어난다고 판단하여 보류하였다. 여섯째, Copilot이 생성한 초안 Zap에는 조건 분기 단계가 포함되어 있지 않았기 때문에, Google Sheets와 Gmail 사이에 Filter by Zapier를 별도로 추가하고 “4점” 또는 “5점”을 포함하는 OR 조건을 설정하여 과제에서 요구하는 조건 분기를 구현하였다.
6. 도구별 비교 분석
UI/UX 측면에서 Make는 캔버스 위에 모듈을 원형 아이콘으로 배치하고 선으로 연결하는 시각적 노드 기반 편집기를 제공하며, Router로 분기할 때 여러 갈래가 화면상에서 물리적으로 뻗어나가는 형태로 표현되어 전체 흐름을 한눈에 파악하기 좋았다. 반면 Zapier는 Trigger부터 Action까지 위에서 아래로 이어지는 세로 리스트 형태의 스텝 카드 기반 편집기를 사용하는데, 순차적인 흐름을 읽기는 쉬우나 Make만큼 분기 구조를 직관적으로 보여주지는 못하였다.
설정 난이도 면에서는, Make는 모듈을 하나하나 직접 검색·추가·연결해야 하고 Router·Filter 위치, 트리거 커서, Limit 값 등 세부 옵션을 사용자가 명확히 이해하고 설정해야 하지만 그만큼 정교한 제어가 가능하였다. Zapier는 Copilot에 자연어 한 줄을 입력하면 Trigger와 Action 초안이 자동으로 생성되어 초기 진입 장벽이 낮았으나, 조건 분기는 자동으로 생성되지 않아 별도로 추가해야 했고 “신규 행만 처리”한다는 트리거의 동작 방식은 실제로 사용해 보아야 체감할 수 있었다.
연동 서비스 범위에서는 두 도구 모두 Google Sheets, Slack, Gmail 연동을 충분히 지원하였으며, 본 과제 범위 내에서는 차이를 체감하기 어려울 만큼 폭넓게 지원되었다. 다만 Zapier는 7,000개 이상의 앱을 지원한다고 안내되는 대신 Functions, Lead router, Transfer data와 같은 고급 데이터 처리 기능은 유료 플랜으로 안내되는 경우가 많았다.
무료 플랜 범위 면에서는 Make가 과제 수행 범위 내에서 막히는 기능이 없었던 반면, Zapier는 기본적인 Trigger‒Filter‒Action 구성은 무료 요금제에서 가능했지만 기존에 쌓인 데이터를 일괄 처리하는 기능들이 제한되어 있어 과제 방향을 조정해야 했다.
실행 로그 확인 방식에서는 Make가 각 모듈 실행 후 초록색 체크로 성공 여부를 즉시 보여주고 Bundle 단위로 입력·출력 데이터와 크레딧 사용량까지 상세하게 확인할 수 있었던 반면, Zapier는 Zap Runs 로그에서 각 스텝의 Data in/out을 확인할 수 있고 필터 통과 여부를 명확한 문구로 보여주었으나 과거 실행 기록 조회 시 접근 제약이 있었다.
트리거 동작 방식에서는 가장 뚜렷한 차이가 나타났는데, Make의 Watch New Rows는 마지막으로 읽은 위치인 커서를 기억하는 방식이어서 Choose where to start 설정으로 전체 재처리 여부를 직접 제어할 수 있었던 반면, Zapier의 New Spreadsheet Row는 Zap이 켜진 시점 이후 새로 추가되는 행만 한 건씩 처리하는 구조로, 기존에 쌓여 있던 데이터는 별도 기능 없이는 자동으로 소급 처리되지 않았다.
7. 적합한 상황에 대한 의견
Make는 여러 갈래로 분기되는 복잡한 워크플로우를 설계하거나 각 단계의 실행 데이터를 세밀하게 검증하며 디버깅해야 하는 경우에 적합하다고 판단된다. 실습 중에도 Router 구조와 Bundle 단위 로그 덕분에 어느 지점에서 몇 건이 통과했는지를 정확히 추적하며 네 단계에 걸친 오류를 순차적으로 해결할 수 있었다. 반면 Zapier는 자동화 도구를 처음 접하거나 빠르게 프로토타입을 만들어보고 싶은 경우에 적합하다. Copilot의 자연어 프롬프트만으로 Trigger·Action 초안을 즉시 얻을 수 있어 일단 동작하는 것부터 만들어보고 다듬는 방식의 작업에 유리하지만, 기존에 쌓인 데이터를 일괄 처리해야 하거나 세밀한 조건 분기가 처음부터 필요한 경우에는 무료 요금제의 기능 제한을 먼저 확인할 필요가 있다.
8. 결론
동일한 “설문 응답 → 조건 분기 → 알림 전송” 워크플로우를 Make와 Zapier 두 도구로 각각 구현한 결과, 두 도구 모두 Trigger‒Filter‒Action 구조를 실제로 동작시킬 수 있었으나 설계 방식과 트러블슈팅 포인트는 뚜렷하게 달랐다. Make는 시각적 분기와 세밀한 로그 제어에 강점이 있었고, Zapier는 Copilot을 통한 빠른 초안 생성에 강점이 있는 대신 조건 분기 추가와 무료 요금제의 기능 제한을 별도로 확인해야 했다. 이번 비교 구현을 통해 동일한 자동화 목표라도 도구의 트리거 동작 방식, 조건 분기 자동 포함 여부, 요금제별 기능 제한이 실제 설계와 트러블슈팅 과정에 직접적인 영향을 미친다는 점을 실습으로 확인할 수 있었으며, 필터를 통과한 응답 10건 전체가 Make에서는 Slack·Gmail 두 액션 모두에, Zapier에서는 Gmail·Slack 두 액션 모두에 실제로 전송되는 것을 최종적으로 확인하였다.
